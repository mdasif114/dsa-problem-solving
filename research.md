# Problem Selection and Practice Method

This document explains how the coding curriculum is selected and reviewed. It is deliberately company-neutral and does not claim to predict any employer's interview questions.

## Selection principles

1. Prefer patterns that recur in production engineering: parsing, aggregation, windows, queues, graphs, scheduling, and bounded state.
2. Prefer problems that allow meaningful follow-ups involving scale, concurrency, reliability, or observability.
3. Separate a representative practice scenario from evidence that a specific question was asked.
4. Treat anonymous reports and informal advice as weak signals, not guarantees.
5. Never use confidential, leaked, proprietary, or employer-specific interview material.
6. Update priorities using actual performance data from progress.md rather than assumptions.

## Classification

- Must: high-value patterns that should be solved and explained fluently.
- Recommended: useful breadth after the must-practise set is stable.
- Additional: targeted coverage for identified weaknesses.
- Representative: selected because it exercises an important engineering pattern.
- Recurring pattern: a technique that appears across many public coding and engineering contexts; it is not a prediction.

## Review method

Each meaningful problem should be reviewed across:

- contract and assumptions;
- correctness and invariant;
- Java implementation quality;
- edge cases and malformed input;
- time and auxiliary-space complexity;
- follow-up changes;
- production implications.

The repository should show honest progress. An incomplete starter is more useful than an untested solution presented as finished.

## Portfolio boundary

Interview schedules, recruiter communications, behavioural answers, company names, and employer-specific preparation belong in private storage. This public-facing repository should contain reusable engineering knowledge only.
