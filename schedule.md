# Weekly Coding Practice Schedule

Use this schedule as a reusable loop rather than a company-specific countdown.

## Session structure

- 10 minutes: recall a previous invariant or mistake
- 40–45 minutes: timed primary problem
- 20 minutes: repair, tests, and edge cases
- 10 minutes: explain the solution aloud
- 5 minutes: update progress.md

## Four-week cycle

| Week | Focus | Expected output |
|---|---|---|
| 1 | Arrays, hashing, parsing, sliding windows | Two completed implementations with tests and complexity notes |
| 2 | Intervals, stacks, queues, heaps, scheduling | Two completed implementations and one follow-up analysis |
| 3 | Trees, graphs, dependencies, idempotency | One completed graph problem and one reliability-oriented case study |
| 4 | Concurrency, streaming, rate limiting, review | One timed mock, one revisit, and a retrospective |

## Spaced review

For each problem:

1. Attempt it without notes.
2. Explain the invariant the next session.
3. Re-solve it after 3–5 days.
4. Repeat under time pressure after 8–12 days if correctness or testing scored below 3.

## Adaptation rules

- Replace new difficulty with a revisit when correctness is below 3.
- Prefer a simpler variant when implementation takes longer than the agreed timebox.
- Record the single most important improvement in progress.md.
- Do not claim completion until the code and tests run successfully.
